# Jane Doe

## About me

- My name is Jane Doe.
- I live in Vienna and usually work in Central European Time.
- I am a product-minded software engineer who likes practical tools, clear writing, and calm execution.
- My primary machine is a MacBook Pro.
- I prefer direct communication, but I do not like rushed decisions.
- I care about privacy, maintainability, accessibility, and making technology easier for regular people.
- I speak English in all communication.
- My main interests are local-first software, thoughtful AI tools, urban cycling, coffee, photography, and learning languages.

## How I like to work

- Be concise first, then add detail when it matters.
- Tell me what you verified and what you are assuming.
- If something is risky, uncertain, or likely to create maintenance work, say so plainly.
- Prefer small, reversible steps over big rewrites.
- When planning, give me the shortest useful plan and include open questions only when they block progress.
- When implementing, follow the existing project style before introducing new patterns.
- I like examples that are realistic and complete enough to copy from.

## Personal preferences

- I usually start work around 09:00 and try to avoid meetings before 10:00.
- I prefer async updates unless a decision is genuinely blocked.
- I like travel plans that leave buffer time and avoid very early departures.
- I eat vegetarian food most of the time.
- I prefer metric units, 24-hour time, and dates written as YYYY-MM-DD when precision matters.
- I like recommendations that include one best option and a short reason.

## Tools I use most

- macOS on a MacBook Pro.
- Zed and VS Code for coding.
- Terminal, zsh, git, GitHub, and Homebrew for daily development.
- Linear and Notion for planning and project notes.
- Figma for product and interface work.
- ChatGPT, Claude Code, Codex, and OpenCode for AI-assisted work.
- Safari for normal browsing and Chrome for testing.

## Coding preferences

- Favor simple, explicit code over clever abstractions.
- Keep changes scoped to the task unless a small cleanup is needed to make the task safe.
- Use the package manager and tooling already used by the project.
- Run the relevant formatter, linter, type checks, and tests before calling work done.
- Add tests for user-facing behavior, bug fixes, and anything that could regress silently.
- Do not add comments unless the code would be difficult to understand without them.
- Avoid broad dependency additions unless they clearly reduce risk or complexity.
- For command-line tools, prioritize predictable output, clear errors, and fast startup.

## Boundaries

- Do not invent facts about me, my work, or my preferences.
- Ask before making destructive changes, deleting data, sending messages, or publishing anything.
- If instructions conflict, point out the conflict and ask for the smallest clarification needed.
- If you cannot verify something important, say that directly instead of guessing.
